### 2. Алгоритм: сквозной пайплайн

Логика оценки метода едина и не зависит от конкретной модели. Один прогон для
одной целевой величины проходит пять стадий:

```
data.xlsx -> [1] загрузка -> [2] синтез -> [3] LOGO-валидация -> [4] метрики -> [5] отчёт
```

**[1] Загрузка** - [core/data/loader.py](../core/data/loader.py).
Чтение таблицы без заголовка, отбор строк с материалом «композитный»/«стальной»,
разбор колонок по позициям в типизированный `DataFrame`. Каждому образцу
присваивается `profile_id` - хэш геометрических признаков (всё, кроме `a/h₀`);
именно по нему далее выделяются группы.

**[2] Синтез образцов** - [core/data/synth.py](../core/data/synth.py).
Для устойчивости моделей к шуму и обеспечения объёма выборки к каждому реальному
образцу добавляется `samples_per_profile` копий с малым равномерным возмущением
входов (`H ±3 мм`, `s ±0.3 мм`, `R ±3 МПа`, `E ±50 МПа`). Целевая величина
$Q_\text{дв}$ **не пересчитывается** (обоснование - раздел 3.2 ТЗ). Синтетические
строки помечаются флагом `is_synth = 1`. Отключается флагом `--no-synth`.

**[3] Leave-One-Group-Out** - [core/evaluation/validation.py](../core/evaluation/validation.py).
Обычное train/test-разбиение неприменимо (завысило бы точность из-за дублей).
Вместо него - по каждому из 6 профилей: обучение на остальных, предсказание
отложенного профиля. Ключевой момент защиты от утечки: в **тест** попадают только
реальные образцы (`~is_synth`), а из **обучения** исключается вся группа целиком
(и реальные, и синтетические её образцы). Так синтетический «клон» отложенного
профиля не может просочиться в обучение.

**[4] Метрики** - [core/evaluation/metrics.py](../core/evaluation/metrics.py).
Считаются только по реальным образцам с конечным предсказанием. Основные (из
методики диссертации): среднее отношение $Q_\text{эксп}/Q_\text{пред}$ (в идеале ~ 1)
и коэффициент вариации `CV`. Дополнительно - `MAE`, `RMSE`, `R²`, `MAPE`,
доля попаданий в +-15 %, доля отрицательных и «в запас» предсказаний.

**[5] Отчёт** - [core/reporting.py](../core/reporting.py).
Сборка сводной таблицы «метод × метрики» (сортировка по `RMSE`), печать в консоль
и - по запросу - экспорт `CSV` + `PNG` (scatter $Q_\text{эксп}/Q_\text{пред}$ и
столбчатое сравнение `RMSE`).

Оркестрирует все пять стадий [core/evaluation/runner.py](../core/evaluation/runner.py) -
функция `run()`, единая точка входа для любого метода. Дополнительно она оценивает
переобучение: `overfit = R²(обучение) − R²(LOGO)`.

### 3. Архитектура: модули и ответственность

Проект разделён на слои; зависимости направлены от точек входа к ядру, ядро о

| **`core/data/`** | слой данных                                                                                                                                                                          |
| [loader.py](../core/data/loader.py) | чтение и чистка xlsx, присвоение `profile_id`.                                                                                                                                       |
| [synth.py](../core/data/synth.py) | синтез возмущённых образцов.                                                                                                                                                         |
| **`core/models/`** | модели                                                                                                                                                                               |
| [base.py](../core/models/base.py) | абстрактный интерфейс `BaseModel` (`fit`/`predict`).                                                                                                                                 |

| [registry.py](../core/models/registry.py) | реестр моделей: `get_model(name)`, `available_models()`. Новый метод регистрируется здесь и сразу доступен во всех точках входа.                                                     |
| **`core/evaluation/`** | оценка                                                                                                                                                                               |
| [validation.py](../core/evaluation/validation.py) | схема Leave-One-Group-Out.                                                                                                                                                           |

| [interactive.py](../entrypoint/interactive.py) | интерактивное меню выбора метода и параметров (демонстрация).                                                                                                                        |

### 4. Упаковка и автоматизация

- **Самодостаточный бинарный файл**.
  PyInstaller собирает интерактивное меню в один файл с зашитыми данными и
  зависимостями (Windows -> `.exe`, Linux -> ELF). Запуск без установленного Python
  - для показа на любой машине.
- **Релизный workflow** - [.github/workflows/release.yml](../.github/workflows/release.yml).
  По пушу тега `v*` GitHub Actions параллельно собирает бинарные файлы на `windows-latest`
  и `ubuntu-22.04` и прикрепляет их к релизу.

### 5. Текущее состояние и переход ко второй части

**Реализовано:** полный скелет пайплайна (стадии 1-5), инфраструктура запуска
(CLI, интерактивный режим, CI) и единственный метод - **линейная регрессия** как
базовая линия.

**Смысл базовой линии:** это планка снизу. Любой более сложный метод, не
превзошедший её, признаётся для данной задачи неоправданным. Поэтому корректная
оценка линейной регрессии - отправная точка для всех дальнейших сравнений.
